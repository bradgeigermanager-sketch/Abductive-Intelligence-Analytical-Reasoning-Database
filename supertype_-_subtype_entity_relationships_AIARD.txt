Once you move to a **supertype/subtype architecture**, the database stops being a collection of records and starts becoming a formal representation of the analyst's world model.

This pattern is used in many intelligence, law enforcement, fraud, cyber, and knowledge-management systems:

```text
ENTITY (supertype)
│
├── PERSON
├── ORGANIZATION
├── LOCATION
├── EVENT
├── DOCUMENT
├── TECHNICAL_OBJECT
├── INFRASTRUCTURE
├── CAMPAIGN
└── DIGITAL_IDENTITY
```

The advantages are:

* Common fields exist once
* Specialized fields exist only where relevant
* New entity types can be added later
* Relationships become universal

***

# Core Supertype

## Entity

Every real-world thing starts here.

```sql
CREATE TABLE entities (

    entity_id BIGSERIAL PRIMARY KEY,

    entity_type VARCHAR(50) NOT NULL,

    canonical_name VARCHAR(500),

    description TEXT,

    confidence_score NUMERIC(5,2),

    first_seen TIMESTAMP,

    last_seen TIMESTAMP,

    status VARCHAR(50),

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Examples:

```text
Person
Organization
Aircraft
IP Address
Malware
Battle
Company
Country
```

# Person Subtype

```sql
CREATE TABLE persons (

    entity_id BIGINT PRIMARY KEY,

    given_name VARCHAR(200),

    middle_name VARCHAR(200),

    family_name VARCHAR(200),

    birth_date DATE,

    nationality VARCHAR(100),

    gender VARCHAR(50),

    deceased BOOLEAN,

    FOREIGN KEY (entity_id)
        REFERENCES entities(entity_id)
);
```

Examples:

```text
Analyst
Threat Actor
CEO
Politician
Pilot
Researcher
```

# Universal Relationship Engine

This becomes the heart of the platform.

```sql
CREATE TABLE entity_relationships (

    relationship_id BIGSERIAL PRIMARY KEY,

    source_entity_id BIGINT NOT NULL,

    target_entity_id BIGINT NOT NULL,

    relationship_type VARCHAR(100) NOT NULL,

    confidence_score NUMERIC(5,2),

    valid_from TIMESTAMP,

    valid_to TIMESTAMP,

    notes TEXT,

    FOREIGN KEY (source_entity_id)
        REFERENCES entities(entity_id),

    FOREIGN KEY (target_entity_id)
        REFERENCES entities(entity_id)
);
```

Examples:

```text
PERSON
    EMPLOYED_BY
ORGANIZATION

PERSON
    OPERATES
ACCOUNT

GROUP
    USES
MALWARE

MALWARE
    COMMUNICATES_WITH
INFRASTRUCTURE

EVENT
    OCCURRED_AT
LOCATION

CAMPAIGN
    TARGETS
ORGANIZATION
```

# Temporal Truth Model

Add temporal validity to nearly everything.

```sql
CREATE TABLE entity_state_history (

    state_id BIGSERIAL PRIMARY KEY,

    entity_id BIGINT,

    state_name VARCHAR(100),

    state_value TEXT,

    effective_from TIMESTAMP,

    effective_to TIMESTAMP,

    FOREIGN KEY(entity_id)
        REFERENCES entities(entity_id)
);
```

Example:

```text
Organization Status
=
Active

2020-01-01 through Present
```

# Intelligence-Centric Extension

Eventually I would add:

```sql
CREATE TABLE collection_requirements
CREATE TABLE information_gaps
CREATE TABLE analytic_questions
CREATE TABLE priority_intelligence_requirements
CREATE TABLE competing_hypotheses
CREATE TABLE deception_indicators
CREATE TABLE cognitive_bias_checks
```

This would allow the platform to track not only:

```text
What we know
```

but also:

```text
What we don't know

What we think

Why we think it

How sure we are

How we might be wrong
```

At that point the schema evolves from a standard entity-resolution system into a true **Entity-Centric Abductive Intelligence Database**, where observations, entities, hypotheses, assumptions, confidence, decisions, and outcomes are all connected within a single relational reasoning framework.
